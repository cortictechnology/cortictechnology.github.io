cortic_platform.sdk.app module¶

class cortic_platform.sdk.app.App¶

Bases: object

The App class is a generic class that developers can extend to create custom applications for the Cortic Platform. To create an app, developers need to implement the following lifecycle event methods:

  1. setup: This method is executed during app initialization. Developers should implement any required initialization and preparation steps here.

  2. process: This method contains the app’s main logic and is executed repeatedly during the app’s lifecycle.

  3. cleanup: This method is executed when the app is shutting down. Developers should implement any necessary cleanup and resource-release steps here.

For each app, an AppManager is responsible for driving and managing the app. It maintains a running loop where the process method is called repeatedly. The App class has an on_exception method for handling exceptions that might occur during the app’s lifecycle. It supports various exception types, and subclasses can override this method to provide custom exception handling.

Variables:

widget_tree (WidgetTree) – The widget tree of the app. It is used to build the UI of the app.

Example:

from cortic_platform import App
from cortic_platform.logging import log, LogLevel
class MyCustomApp(App):
    def setup(self):
        # Implement any required initialization and preparation steps here.
        pass
    def process(self):
        # Implement the app's core logic here.
        pass
    def cleanup(self):
        # Implement any necessary cleanup and resource-release steps here.
        pass
    def on_exception(self, exception, exception_data=None):
        # Implement custom exception handling here.
        log("An exception occurred: " + str(exception), LogLevel.Error)

abstract setup()¶

Abstract Method. Called during app initialization. Implement any required initialization and preparation steps here. For example, if the app requires a camera, the camera should be initialized here. This makes it easier for the app to handle exceptions that might occur during initialization. It also makes it easier for developers to implement the cleanup method to release resources. For example, the camera can be released in the cleanup method. This method is called only once during the app’s lifecycle. It is not recommanded to perform any computationally intensive tasks in this method.

Raises:

NotImplementedError: If the method is not implemented in the subclass.

abstract process()¶

Abstract Method. Called repeatedly in every step of the app’s main event loop. This method contains the app’s main logic. The number of times this method is called per second is bounded by the maxFPS setting within the app’s manifest file. Since the method is called repeatedly, it is not recommended to perform any operations related to resource acquisition or release in this method as it might cause performance issues.

Raises:

NotImplementedError: If the method is not implemented in the subclass.

abstract cleanup()¶

Abstract method for cleaning up the application and releasing resources. This method is called when the app is shutting down. Implement any necessary cleanup and resource-release steps here. For example, if the app requires a camera, the camera should be released here.

Raises:

NotImplementedError: If the method is not implemented in the subclass.

on_exception(exception, exception_data=None)¶

A method for handling exceptions in the application. This method can be overridden by subclasses to provide custom exception handling. Below is a list of exception types that can be handled by this method. The most common exception types are ProcessError, NotSchedulable, and WidgetBuildFailed. The other exception types are related to the app’s lifecycle and are less common. ProcessError can occur when the app’s process method encounters an error. Developers should implement custom exception handling for this exception type as the app’s main logic is implemented in the process method. NotSchedulable can occur when a node is not schedulable during the start of a pipeline. Developers can decide to change the pipeline’s configuration so that the node is schedulable on a different device. WidgetBuildFailed can occur when the app’s UI widget fails to build. This can happen when the app’s UI widget is not configured correctly. No matter what exception type is encountered, developers can decide if they want to continue running the app or stop it. It is recommended to log the exception and shut down the app if the exception is critical. If the app is stopped, the app’s cleanup method is called. Otherwise, the app’s process method is called again.

Exception Type

Description

ExceptionTypes.SetupError

Errors occurring during the setup phase.

ExceptionTypes.ProcessError

Errors occurring during the process phase.

ExceptionTypes.TeardownError

Errors occurring during the teardown phase.

ExceptionTypes.NotSchedulable

Errors related to a node not being schedulable.

ExceptionTypes.NodeExpired

Errors related to a node scheduling is expired.

ExceptionTypes.CriticalError

Critical errors that might require immediate action.

ExceptionTypes.WidgetBuildFailed

Errors occurring during the building of a UI widget.
Parameters:

  • exception (ExceptionTypes) – The exception type that occurred. in different colors in the Hub’s console widget.

  • exception_data (Optional[Any]) – Additional data related to the exception.

cortic_platform.sdk.pipeline module¶

The pipeline module provides a framework for creating and managing data processing pipelines using the Pipeline and PipelineNode classes. These classes are designed to simplify the process of connecting and orchestrating multiple services in a directed acyclic graph structure. The module allows developers to build complex data processing pipelines with ease, while managing the flow of data, relationships between nodes, and association of nodes to services.

Main components of the pipeline module:

  • PipelineNode

  • Pipeline

By using the pipeline module, developers can create efficient and modular data processing pipelines, combining different services in a flexible and intuitive manner.

class cortic_platform.sdk.pipeline.PipelineNode(service_handle, device_name='')¶

Bases: object

A PipelineNode represents a node in an a Pipeline object, associated with a specific service. Developers can connect multiple nodes to create an execution pipeline for the services associated with these nodes, maximizing the throughput of each service. To use a PipelineNode, it must be added to a Pipeline instance.

Variables:

  • state (PipelineNodeState) –

    The current state of the node. This is automatically updated by a Pipeline. The possible states are:

    • PipelineNodeState.NotAssociated: The node is not associated with any service.

    • PipelineNodeState.Associating: The node is currently trying to associate with a service.

    • PipelineNodeState.Associated: The node is associated with a service.

    • PipelineNodeState.FailedToAssociate: The node failed to associate with a service.

  • device_name (str) – The name of the device that the node requests the service from. This is specified when the node is created. If the device name is not specified, the node will request the service from any device that provides it.

  • associated_device (dict) – The device that the node is associated with. This is automatically updated by a Pipeline.

  • associated_service (dict) – The service that the node is associated with. This is automatically updated by a Pipeline.

  • associated_service_states (dict) – A dictionary of the states of the services associated with the node.

  • input_nodes (dict) – A dictionary of the input nodes of the node. The keys are the names of the input ports of the service associated with this node, and the values are dictionaries of containing the names of input data and the nodes from which to receive it. The name of input ports and input data must match the names specified in the service’s implementation.

  • output_nodes (list) – A list of the output nodes of the node. The output nodes are the nodes that receive the output of the service associated with this node.

__init__(service_handle, device_name='')¶
Parameters:

  • service_handle (str) –

    The handle name of a specific service that the developer needs to invoke. This name is defined by the developer when they create or import the service into an App. The service associated with the handle can be easily swapped with another service performing the same function at runtime.

    Example:

    from cortic_platform import PipelineNode
webcam_service = PipelineNode("face_detection_service")

    In the above example, only the service_handle is provided. When the pipeline is started, the Cortic Platform Runtime will automatically search and associate a “face_detection_service” service within the local network to this node. If the node crashes or disappears anytime during pipeline execution, a fail-over mechanism will try to associate another “face_detection_service” service to this node. If no such service is found, an exception is thrown. The “face_detection_service” handle is swappable, meaning that the PipelineNode can easily associate with different services performing the same function. This enables developers to seamlessly switch between various service implementations for better performance, updated features, or alternative approaches to the same problem without changing the pipeline code.

  • device_name (str, optional) –

    The device that the user wants the service from. A device is a hardware resource that exists in the user’s local network. Users can obtain a list of these devices from the App’s device_list variable.

    Example:

    from cortic_platform import PipelineNode
webcam_service = PipelineNode("webcam_capture", device_name="michael_pc")

    In the above example, with the provided device_name, the Cortic Platform Runtime will only search and associate a “webcam_capture” service from the device “michael_pc” to this node. It is important to note that “webcam_capture” services may not be swappable in most cases, as different webcams may be capturing different views. To ensure the correct webcam is used for the specific node, it is recommended to provide the device_name to avoid any automatic fail-over to an undesired webcam service. In a fail-over situation, the Cortic Platform Runtime will only try to re-associate a “webcam_capture” service from the device “michael_pc” to this node. If no such service is found, an exception is thrown. If the device “michael_pc” is not found, an exception is thrown as well. By specifying the device_name, developers can ensure that their desired webcam service is used, preventing unexpected behavior due to automatic fail-over to another webcam service.

set_input(input_map: dict)¶

Configures the service associated with this node to listen to the output channel(s) of the services associated with the provided nodes for input data.

By default, services associated with the nodes in the list are not activated and do not produce any output. When a node is added to a Pipeline and the Pipeline is started, the input services for that node are activated and begin sending their outputs to their respective output channels. Conversely, if a service’s output is not required by any other nodes, the service will be deactivated to save resources. This design ensures efficient resource usage by only processing and transmitting data when needed.

Parameters:

input_map (dict) – A dictionary of input data sources. The keys are the names of the input ports of the service associated with this node, and the values are dictionaries of containing the names of input data and the nodes from which to receive it. The name of input ports and input data must match the names specified in the service’s implementation.

Example:

from pipeline import PipelineNode
webcam_node = PipelineNode("webcam_service")
facemesh_estimation_node = PipelineNode("facemesh_estimation_service")
facemesh_estimation_node.set_input({"camera_input": {"image": webcam_node}})
class cortic_platform.sdk.pipeline.Pipeline(name)¶

Bases: object

The Pipeline class represents a data processing pipeline consisting of a set of PipelineNodes. A pipeline can be thought of as a directed acyclic graph where each node represents a step in the pipeline. The class manages the relationships between nodes, the flow of data through the pipeline, and the association of nodes to services. Developers can retrieve the latest result of any service during the App’s process loop and start or stop a pipeline at any time. Each PipelineNode must be added to a Pipeline instance to be used in the data processing pipeline.

Variables:

  • name (str) – The name of the pipeline.

  • nodes (list) – A list of PipelineNodes that are part of this pipeline.

  • input_ports (dict) – A dictionary of input ports. The keys are the names of the input ports, and the values are the data types of the input ports.

  • output_ports (dict) – A dictionary of output ports. The keys are the names of the output ports, and the values are the nodes that are associated with the output ports.

  • state (PipelineState) – The current state of the pipeline.

Example:

from cortic_platform import Pipeline, PipelineNode
pipeline = Pipeline("example_pipeline")
input_node = PipelineNode("input_node")
processing_node = PipelineNode("processing_node")
processing_node.set_input({"input": {"data": input_node}})
output_node = PipelineNode("output_node")
output_node.set_input("result": {"data": processing_node})
pipeline.add_node(input_node)
pipeline.add_node(processing_node)
pipeline.add_node(output_node)
pipeline.add_output_port("output_name", output_node)
pipeline.start()

class cortic_platform.sdk.service.ServiceContext¶

Bases: object

The ServiceContext class provides a set of methods that can be used by a Service to manage its states. The ServiceContext class is automatically instantiated when a Service object is created. The ServiceContext object is passed to the Service object

  1. get_state: Retrieves the value of a state.

  2. set_state: Sets the value of a state.

  3. reset_states: Resets all states to their default values.

  4. clear_states: Clears the entire state dictionary. Developers need to calle the create_state() method

    to create new states after calling this method.

A state is a variable that can be used by a Service to store information that is required by the Service to perform its task. For example, a Service may need to store the number of times it has processed data. In this case, the Service can create a state called “count” and increment its value each time it processes data.

class cortic_platform.sdk.service.Service¶

Bases: object

The Service class is a versatile foundation for developers to build custom services that can be used in a Cortic AIoT (Artificial Intelligence of Things) App. To create a service, developers should implement the following methods, which represent key stages in the service lifecycle:

  1. activate: In this stage, the service should gather all required resources.

  2. process: This method contains the data processing logic for the service and is executed when new data arrives from an App or another Service.

  3. deactivate: During this stage, the service should release all previously acquired resources.

A service is activated only when an App or another Service requires its output. It will be automatically deactivated once its output is no longer needed. By releasing resources during the deactivation process, the system ensures that resources are always available to the services that genuinely need them.

__init__()¶

Initializes a service

abstract activate()¶

Abstract Method. Allocates and acquires any resources required by this service. This method is executed when an App or another Service requires the output of this Service. Developers should implement this method to acquire any resources that are required by the process() method. The resources acquired in this method should be released in the deactivate() method. For example, if the process() method requires a neural network model, the activate() method should load the model from disk and store it in a variable. The deactivate() method should release the model by deleting the variable. This ensures that the model is loaded only when it is required and is released when it is no longer needed.

Another thing developers should do in this method is to setup the context of this service. The context is a dictionary that can be used to store state information that is required by the service to perform its task. For example, a service may need to store the number of times it has processed data. In this case, the service can create a state called “count” and increment its value each time it processes data. However, the context is reset each time the service is deactivated and activated again. Therefore, if the service needs to maintain a state, it should create the state in the init() method instead.

Raises:

NotImplementedError: If this method is not implemented.

abstract process(input_data=None)¶

Abstract Method. Processes the input data and produces an output. This method is executed when new data arrives from an App or another Service. There isn’t a fixed number of calls per second for this method. The number of calls per second depends on the rate at which data is produced by the App or Service that is connected to this Service. Developers should implement this method to process the input data and produce an output. The output of this method is returned to the App or Service that is connected to this Service. It is not recommended to perform any resource allocation in this method. All resources should be allocated in the activate() method. This method should only contain the data processing logic.

Parameters:

input_data (dict) – Contains the data that needs to be processed by this Service. The data are provided as <key>-<value> pairs, in which the <key> is the name of the data, and <value> is the value of this data.

Returns:

The output of this Service.

Return type:

dict

Raises:

NotImplementedError: If this method is not implemented.

abstract deactivate()¶

Abstract Method. Releases any resources being allocated and obtained by this Service in the activate() method. This method is executed when an App or another Service no longer requires the output of this Service. Developers should implement this method to release any resources that are no longer required.

Raises:

NotImplementedError: If this method is not implemented.

class cortic_platform.sdk.streaming_peer.StreamingPeer(name, incoming_connection_handler)¶

Bases: object

The StreamingPeer class facilitates streaming data between different Cortic Apps. It can both send and receive data streams for further processing or visualization. This class leverages WebRTC (Web Real-Time Communication) to establish connections and exchange data like video, audio, and other information. Developers can visualize the data stream using the StreamingView widget in their Cortic Apps, as well as capture any mouse and keyboard events generated by users interacting with the stream.

Variables:

  • outgoing_connections (list) – A list of StreamingPeerConnection objects that define the outgoing streams.

  • incoming_connections (list) – A list of StreamingPeerConnection objects that define the incoming streams.

  • state (StreamingPeerState) – The state of the StreamingPeer. It can be one of the following values: AwaitIncomingConnection, RefuseIncomingConnection, NotAbleToSignal, NetworkError.

__init__(name, incoming_connection_handler)¶

Initializes a StreamingPeer with necessary callbacks.

Parameters:

  • name (str) – The name of the StreamingPeer. This name will be used to identify the StreamingPeer in the Cortic App.

  • on_connection_request (function) –

    A callback function executed when another StreamingPeer requests a connection. This function should return a dictionary that outlines the routing rules for incoming tracks. The dictionary keys denote incoming track IDs, and the values are lists of tuples. Each tuple contains two elements: first, an instance of a Python class implementing the update_data method from the DataConsumer interface; second, a list of strings representing the input names of the DataConsumer. These tuples specify the destinations for routing incoming tracks. If no routing is necessary, return an empty dictionary.

    Example:

    class VideoConsumer:
    def __init__(self):
        self.data = {"input_1": None}

    def update_data(self, input_name, data):
        self.data[input_name] = data

video_consumer = VideoConsumer()

def on_connection_request(peer_name):
    return {"video_track_1_id": [(video_consumer: ["input_1"])],
            "audio_track_1_id": [(my_widget, ["audio"])]}

  • on_connected (function) – A callback function that will be called when connection is established with another StreamingPeer.

  • on_connection_failed (function) – A callback function that will be called when connection is failed with another StreamingPeer.

  • on_disconnected (function) – A callback function that will be called when connection is disconnected with another StreamingPeer.

Raises:

  • ValueError – If the name of the StreamPeer already exists.

  • TypeError – If the tuple in the routing rules does not contain proper elements. i.e. the first element is not a DataConsumer instance, and the second element is not a list of strings.

  • ValueError – If the input names in the routing rules do not exist in the DataConsumer.

Example:

from cortic_platform import StreamingPeer

class VideoConsumer:
    def __init__(self):
        self.data = {"input_1": None}

    def update_data(self, input_name, data):
        self.data[input_name] = data

video_consumer = VideoConsumer()

def on_connection_request(peer_name):
    return {"video_track_1_id": [(video_consumer: ["input_1"])],
            "audio_track_1_id": [(my_widget, ["audio"])]}

def on_connected(peer_name):
    print("Connected to peer {}".format(peer_name))

def on_connection_failed(peer_name):
    print("Connection failed with peer {}".format(peer_name))

def on_disconnected(peer_name):
    print("Disconnected from peer {}".format(peer_name))

peer = StreamingPeer("peer1",
                    on_connection_request)
peer.connection_success_handler = on_connected
peer.connection_failed_handler = on_connection_failed
peer.connection_closed_handler = on_disconnected
