class cortic_platform.sdk.app.App

Bases: object

The App class is a generic class that developers can extend to create custom applications for the Cortic Platform. To create an app, developers need to implement the following lifecycle event methods:

  1. setup: This method is executed during app initialization. Developers should implement any required initialization and preparation steps here.

  2. process: This method contains the app’s main logic and is executed repeatedly during the app’s lifecycle.

  3. cleanup: This method is executed when the app is shutting down. Developers should implement any necessary cleanup and resource-release steps here.

For each app, an AppManager is responsible for driving and managing the app. It maintains a running loop where the process method is called repeatedly. The App class has an on_exception method for handling exceptions that might occur during the app’s lifecycle. It supports various exception types, and subclasses can override this method to provide custom exception handling.

Parameters:

  • app_manager (AppManager) – The AppManager instance that manages this App.

  • device_list (dict) – The latest device map received from the Cortic Platform Runtime.

  • ui (UserInterface) – Developer can use this to contruct the user interface of the App.

Example:

from cortic_platform import App
from cortic_platform.logging import log, LogLevel
class MyCustomApp(App):
    def setup(self):
        # Implement any required initialization and preparation steps here.
        pass
    def process(self):
        # Implement the app's core logic here.
        pass
    def cleanup(self):
        # Implement any necessary cleanup and resource-release steps here.
        pass
    def on_exception(self, exception, exception_data=None):
        # Implement custom exception handling here.
        log("An exception occurred: " + str(exception), LogLevel.Error)

cortic_platform.sdk.pipeline module

The pipeline module provides a framework for creating and managing data processing pipelines using the Pipeline and PipelineNode classes. These classes are designed to simplify the process of connecting and orchestrating multiple services in a directed acyclic graph structure. The module allows developers to build complex data processing pipelines with ease, while managing the flow of data, relationships between nodes, and association of nodes to services.

Main components of the pipeline module:

  • PipelineNode

  • Pipeline

By using the pipeline module, developers can create efficient and modular data processing pipelines, combining different services in a flexible and intuitive manner.

class cortic_platform.sdk.pipeline.PipelineNode(service_handle, device_name='')

Bases: object

A PipelineNode represents a node in an a Pipeline object, associated with a specific service. Developers can connect multiple nodes to create an execution pipeline for the services associated with these nodes, maximizing the throughput of each service. To use a PipelineNode, it must be added to a Pipeline instance.

Variables:

  • service_handle (str) – The handle name of a specific service that developer needs to invoke.

  • device_name (str) – The device that the user wants the service from.

  • state (PipelineNodeState) –

    The current state of the node. This is automatically updated by a Pipeline. The possible states are:

    • PipelineNodeState.NotAssociated: The node is not associated with any service.

    • PipelineNodeState.Associating: The node is currently trying to associate with a service.

    • PipelineNodeState.Associated: The node is associated with a service.

    • PipelineNodeState.FailedToAssociate: The node failed to associate with a service.

  • input_nodes (list) – A list of input nodes that are connected to this node.

  • output_nodes (list) – A list of output nodes that depend on the output data of this node.

__init__(service_handle, device_name='')
Parameters:

  • service_handle (str) –

    The handle name of a specific service that the developer needs to invoke. This name is defined by the developer when they create or import the service into an App. The service associated with the handle can be easily swapped with another service performing the same function at runtime.

    Example:

    from cortic_platform import PipelineNode
webcam_service = PipelineNode("face_detection_service")

    In the above example, only the service_handle is provided. When the pipeline is started, the Cortic Platform Runtime will automatically search and associate a “face_detection_service” service within the local network to this node. If the node crashes or disappears anytime during pipeline execution, a fail-over mechanism will try to associate another “face_detection_service” service to this node. If no such service is found, an exception is thrown. The “face_detection_service” handle is swappable, meaning that the PipelineNode can easily associate with different services performing the same function. This enables developers to seamlessly switch between various service implementations for better performance, updated features, or alternative approaches to the same problem without changing the pipeline code.

  • device_name (str, optional) –

    The device that the user wants the service from. A device is a hardware resource that exists in the user’s local network. Users can obtain a list of these devices from the App’s device_list variable.

    Example:

    from cortic_platform import PipelineNode
webcam_service = PipelineNode("webcam_capture", device_name="michael_pc")

    In the above example, with the provided device_name, the Cortic Platform Runtime will only search and associate a “webcam_capture” service from the device “michael_pc” to this node. It is important to note that “webcam_capture” services may not be swappable in most cases, as different webcams may be capturing different views. To ensure the correct webcam is used for the specific node, it is recommended to provide the device_name to avoid any automatic fail-over to an undesired webcam service. In a fail-over situation, the Cortic Platform Runtime will only try to re-associate a “webcam_capture” service from the device “michael_pc” to this node. If no such service is found, an exception is thrown. If the device “michael_pc” is not found, an exception is thrown as well. By specifying the device_name, developers can ensure that their desired webcam service is used, preventing unexpected behavior due to automatic fail-over to another webcam service.

set_input(input_map: dict)

Configures the service associated with this node to listen to the output channel(s) of the services associated with the provided nodes for input data.

By default, services associated with the nodes in the list are not activated and do not produce any output. When a node is added to a Pipeline and the Pipeline is started, the input services for that node are activated and begin sending their outputs to their respective output channels. Conversely, if a service’s output is not required by any other nodes, the service will be deactivated to save resources. This design ensures efficient resource usage by only processing and transmitting data when needed.

Parameters:

input_map (dict) – A dictionary of input data sources. The keys are the names of the input ports of the service associated with this node, and the values are dictionaries of containing the names of input data and the nodes from which to receive it. The name of input ports and input data must match the names specified in the service’s implementation.

Example:

from pipeline import PipelineNode
webcam_node = PipelineNode("webcam_service")
facemesh_estimation_node = PipelineNode("facemesh_estimation_service")
facemesh_estimation_node.set_input({"camera_input": {"image": webcam_node}})
class cortic_platform.sdk.pipeline.Pipeline(name)

Bases: object

The Pipeline class represents a data processing pipeline consisting of a set of PipelineNodes. A pipeline can be thought of as a directed acyclic graph where each node represents a step in the pipeline. The class manages the relationships between nodes, the flow of data through the pipeline, and the association of nodes to services. Developers can retrieve the latest result of any service during the App’s process loop and start or stop a pipeline at any time. Each PipelineNode must be added to a Pipeline instance to be used in the data processing pipeline.

Variables:

  • name (str) – The name of the pipeline.

  • nodes (list) – A list of nodes (PipelineNode instances) in the pipeline.

  • input_ports (dict) – A dictionary mapping input port names to their corresponding input ports.

  • output_ports (dict) – A dictionary mapping output port names to their corresponding output ports. Each output port is associated with a PipelineNode.

  • state (PipelineState) – The current state of the pipeline (Stopped, Running, or FailedToStart).

Example:

from cortic_platform import Pipeline, PipelineNode
pipeline = Pipeline("example_pipeline")
input_node = PipelineNode("input_node")
processing_node = PipelineNode("processing_node")
processing_node.set_input({"input": {"data": input_node}})
output_node = PipelineNode("output_node")
output_node.set_input("result": {"data": processing_node})
pipeline.add_node(input_node)
pipeline.add_node(processing_node)
pipeline.add_node(output_node)
pipeline.add_output_port("output_name", output_node)
pipeline.start()

class cortic_platform.sdk.service.Service

Bases: object

The Service class is a versatile foundation for developers to build custom services that can be used in a Cortic Platform App. To create a service, developers should implement the following methods, which represent key stages in the service lifecycle:

  1. activate: In this stage, the service should gather all required resources.

  2. process: This method contains the data processing logic for the service and is executed when new data arrives from an App or another Service.

  3. deactivate: During this stage, the service should release all previously acquired resources.

A service is activated only when an App or another Service requires its output. It will be automatically deactivated once its output is no longer needed. By releasing resources during the deactivation process, the system ensures that resources are always available to the services that genuinely need them.

This class leverages WebRTC (Web Real-Time Communication) to establish connections and exchange data like video, audio, and other information. __init__(name, on_connection_request, on_connected=None, on_connection_failed=None, on_disconnected=None)

Initializes a StreamingPeer with necessary callbacks.

Parameters:

  • name (str) – The name of the StreamingPeer. This name will be used to identify the StreamingPeer in the Cortic App.

  • on_connection_request (function) –

    A callback function executed when another StreamingPeer requests a connection. This function should return a dictionary that outlines the routing rules for incoming tracks. The dictionary keys denote incoming track IDs, and the values are lists of tuples. Each tuple contains two elements: first, an instance of a Python class implementing the update_data method from the DataConsumer interface; second, a list of strings representing the input names of the DataConsumer. These tuples specify the destinations for routing incoming tracks. If no routing is necessary, return an empty dictionary.

    Example:

make_connection_request(peer_name, video_sources=[], audio_sources=[], data_sources=[])

Initiates a connection request to another peer.

Parameters:

  • peer_name (str) – The name of the peer to establish a connection with.

  • video_sources (list) –

    A list of tuples specifying the data sources for video tracks to be sent to another peer. Each tuple contains three elements:

    1. an instance of a Python class implementing the get_data method from the DataProducer interface,

    2. a list of strings representing the output names of the DataProducer, and

    3. an optional string containing the name of a peer.

    When the third element is provided, only data produced by the input data from the specified peer is used in a video track.

  • audio_sources (list) –

    A list of tuples specifying the data sources for audio tracks to be sent to another peer. Each tuple contains three elements:

    1. an instance of a Python class implementing the get_data method from the DataProducer interface,

    2. a list of strings representing the output names of the DataProducer, and

    3. an optional string containing the name of a peer.

    When the third element is provided, only data produced by the input data from the specified peer is used in a audio track.

  • data_sources (list) –

    A list of tuples specifying the data sources to send to datachannels of another peer. Each tuple contains three elements:

    1. an instance of a Python class implementing the get_data method from the DataProducer interface,

    2. a list of strings representing the output names of the DataProducer, and

    3. an optional string containing the name of a peer.

    When the third element is provided, only data produced by the input data from the specified peer is used in a datachannel.

Raises:

  • ValueError – If a connection with the peer already exists.

  • TypeError – If the tuple in video_sources, audio_sources, or data_sources does not contain proper elements. i.e. the first element is not a DataProducer, the second element is not a list of strings, or the third element is not a string.

  • ValueError – If the output names in the tuple in video_sources, audio_sources, or data_sources do not exist in the DataProducer.
